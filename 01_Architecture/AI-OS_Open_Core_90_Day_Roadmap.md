# AI-OS Open-Core 90-Day Roadmap

Version: 1.0.0
Status: Draft
Layer: Architecture
Document Type: Guide
Document ID: DOC-ARCH-009
Owner: AI-OS Architecture

---

## Purpose

This roadmap converts the Open-Core commercial blueprint into an executable 90-day operating plan.

It defines milestones, deliverables, quality gates, ownership model, and measurable success criteria required to launch the first monetizable AI-OS offer.

---

## Scope

Included:

- 90-day milestone plan for productization and commercialization;
- required outputs for Open Core, Pro Packs, and service entry offer;
- decision gates and KPI targets for each phase.

Excluded:

- legal and procurement implementation details;
- detailed technical implementation of platform adapters;
- long-term roadmap beyond day 90.

---

## 1. Execution Principles

1. Open Core SHALL remain adoption-first and publicly reusable.
2. Paid value SHALL be delivered through operational depth, governance assurance, and deployment acceleration.
3. Every phase SHALL end with a measurable decision gate: continue, adjust, or stop.
4. Pilot feedback SHALL be converted into packaged assets, not one-off consulting artifacts.

---

## 2. Milestone Plan

| Phase | Days | Primary Goal | Exit Criteria |
| ----- | ---- | ------------ | ------------- |
| Phase 1 | 1-30 | Product boundary and offer definition | Core/Pro boundaries approved; 2 paid packs scoped; diagnostic offer defined |
| Phase 2 | 31-60 | Pilot validation with real teams | 2-3 pilots completed; friction log collected; pack updates released |
| Phase 3 | 61-90 | Initial commercial launch | Repeatable diagnostic-to-pack flow active; first paid conversions recorded |

---

## 3. Phase Deliverables

### 3.1 Phase 1 (Days 1-30): Offer Definition

Required deliverables:

- Open Core boundary spec (what is free vs paid).
- Governance Pack v1 scope.
- PromptOps Pack v1 scope.
- Diagnostic service statement of work and output template.
- Pilot qualification checklist.

Decision gate:

- The phase passes only if package boundaries are explicit and non-overlapping.

### 3.2 Phase 2 (Days 31-60): Pilot Validation

Required deliverables:

- 2-3 design-partner pilot engagements.
- Pilot findings log (architecture blockers, adoption blockers, packaging gaps).
- Updated Governance Pack v1.1 and PromptOps Pack v1.1.
- Baseline KPI snapshot from pilot execution.

Decision gate:

- The phase passes only if pilots show repeatable value and at least one clear paid conversion path.

### 3.3 Phase 3 (Days 61-90): Launch Readiness

Required deliverables:

- Commercial catalog (offer ladder: diagnostic -> pack deployment -> expansion).
- Delivery playbook for onboarding and pack activation.
- Release cadence definition for paid modules.
- Launch KPI dashboard.

Decision gate:

- The phase passes only if the team can run the same sales-and-delivery workflow without custom redesign.

---

## 4. KPI Targets (Day 90)

| KPI | Target |
| --- | ------ |
| Design-partner pilots completed | 2-3 |
| Pilot-to-paid conversion | >= 30% |
| Time-to-first-value | <= 14 days from kickoff |
| Pack reuse ratio | >= 60% standardized assets vs custom work |
| Expansion readiness | At least 1 account ready for second paid pack |

---

## 5. Risk Register

| Risk | Signal | Mitigation |
| ---- | ------ | ---------- |
| Over-customization in pilots | Each pilot requires unique structure | Force packaging review after every pilot sprint |
| Weak paid differentiation | Prospects use only Open Core | Add governance controls and operational assets only available in paid packs |
| Slow buyer confidence | Long decision cycles | Lead with measurable diagnostic outputs and architecture scorecards |

---

## 6. Ownership Model

| Workstream | Responsible Role | Output |
| ---------- | ---------------- | ------ |
| Offer architecture | AI-OS Architecture Lead | Pack boundaries and standards |
| Pilot delivery | Solution Architect | Pilot execution and findings |
| Productization | PromptOps Owner | Reusable module assets |
| Commercial motion | GTM Owner | Pipeline and conversion process |
